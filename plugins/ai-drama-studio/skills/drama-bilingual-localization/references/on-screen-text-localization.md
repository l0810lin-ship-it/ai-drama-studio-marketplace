# 屏幕文字与短信本地化

## 先建台账，再翻译

逐集扫描对白、动作行、镜头说明、分镜、道具清单和画面 OCR。每一处观众能够读到且影响理解的文字都建立独立 ID，不要只搜索“短信”二字。

至少识别：

- SMS、iMessage、聊天气泡、群聊、私信、发送失败、撤回和已读状态；
- 联系人名称、来电显示、未接来电、未读数量、语音消息标签和通话记录；
- 推送通知、警报、系统提示、App 按钮和剧情相关 UI；
- 社媒正文、配文、评论、直播标题/状态和话题标签；
- 邮件主题/正文、信件、日记、合同、报告、新闻标题、招牌和手写字条；
- 角色读出上述文字时对应的 OS（画面文字）与 VO（声音）。

使用 `assets/screen-text-ledger.csv` 记录来源位置、媒介、发送者/接收者、源文、译文、版面限制、世界观依据和审核状态。

## 翻译规则

1. 先保留剧情功能：身份揭露、时间、地点、威胁条件、关系称谓、证据和钩子不能丢。
2. 再匹配媒介：聊天可以碎片化，系统通知要像系统，新闻标题要像标题，合同不得写成口语。
3. 再匹配角色：年龄、职业、阶层、亲疏、情绪和刻意伪装都要反映在用词、标点、缩写与表情中。
4. 再压缩版面：优先去冗词、改短句、拆气泡；不得通过删剧情信息来硬塞画面。
5. 保留交互元数据：发送者、接收者、时间、排序、换行、表情、已读/未读、撤回、发送失败和附件状态。
6. 联系人昵称属于叙事信息。既要让目标观众立即理解，也要保留关系含义；音译姓名与关系昵称分开处理。
7. OS 与 VO 同时出现时分别登记。若 VO 是逐字朗读，可按交付规格避免字幕机械重复，但不能删除任一资产需求。
8. 原文看不清、缺页或仅有占位 UI 时标 `unreadable` 或 `decorative`，不得编造具体内容。

## 世界观与渠道检查

翻译前回答：这个时代有这种设备吗？这个角色有权限使用吗？这个组织会通过该渠道谈这件事吗？这个媒介是否会暴露剧情逻辑？

- 若年代不支持手机、短信或社媒，标为 `world_conflict`，回到剧本修订流程，不要自行“翻成一封短信”。
- 若世界观要求敏感事务当面谈、使用一次性手机或心灵链接，普通短信中的机密内容要标冲突。
- 若屏幕文字来自角色伪装、诱饵、匿名账户或假联系人，译文必须保留这种欺骗功能。

## 交付与 QA

- 台账中每条记录只能处于 `translated`、`approved_exception`、`decorative`、`unreadable`、`blocked` 之一。
- 检查源文台账条数与译文资产条数是否对齐，逐集报告遗漏数。
- 检查姓名、术语、大小写、标点、电话号码/日期格式、时间格式和换行。
- 检查屏幕截图或版式预览，确认气泡、按钮、道具和字幕没有溢出。
- 新版本出现或删除屏幕文字时，连同对白一起进入增量差异报告。
